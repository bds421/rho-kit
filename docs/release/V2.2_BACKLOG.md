# v2.2+ Backlog

Captured scope for follow-up items that are NOT in v2.0.0 but
have a concrete shape worth recording so a future session can
pick them up with full context. Items here are intentional
deferrals — each was scoped during the v2.0.0 prep sweep and
judged to need its own dedicated wave for quality, not because
the scope is speculative.

## SQS + Cloud Pub/Sub messaging backends

**Why deferred:** Each backend is a multi-thousand-LOC
implementation (NATS reference: 1100 LOC + 700 LOC test). The
v2.0.0 wave budget could not absorb two more backends without
diluting quality on the rest of the wave 157–175 work.

**Scope:**

- `infra/messaging/sqsbackend/` — new module satisfying
  `messaging.Connector` / `messaging.Consumer` / `messaging.Publisher`.
  - Long-polling consumer with `ReceiveMessage` batched up to 10.
  - Visibility-timeout extension for handlers that exceed the
    initial window.
  - Dead-letter routing via SQS RedrivePolicy (DLQ ARN).
  - Topology: queue name = exchange; consumer group is the same
    queue (SQS has no fan-out).
  - Tracing helper: `sqstracing/` with `Carrier` over
    `MessageAttributes` for W3C trace context.
  - Bounded-cardinality metrics: `sqs_messages_consumed_total`,
    `sqs_messages_published_total`, `sqs_handler_duration_seconds`.
- `infra/messaging/pubsubbackend/` — new module for GCP Pub/Sub.
  - Streaming pull (`SubscribeStreaming`) per subscription.
  - Modack flow for handlers that take longer than the ack
    deadline.
  - Dead-letter topic routing via Pub/Sub
    `DeadLetterPolicy`.
  - Topology: topic = exchange; subscription = consumer group.
  - Tracing helper: `pubsubtracing/` with `Carrier` over
    `Attributes`.
  - Bounded-cardinality metrics matching the SQS shape.

**Constraints:**

- Both backends MUST satisfy the kit's
  `make check-dependency-allowlist` and
  `make check-dependency-boundaries` gates — heavy SDK
  dependencies (`github.com/aws/aws-sdk-go-v2/service/sqs`,
  `cloud.google.com/go/pubsub`) stay confined to the backend's
  own module.
- Both MUST emit consume-side metrics with the
  `WithOpaqueConsumeLabels` projection from wave 158 so
  per-tenant topic / subscription names cannot blow up
  Prometheus cardinality.
- Both MUST honour `BindingSpec.WithoutRetry=true` semantics
  matching the kafkabackend precedent: no per-message TTL or
  delayed-redelivery primitive maps cleanly to SQS / Pub/Sub
  ack semantics, so consumer-side retry via `resilience/retry`
  is the documented path.

**Estimated size:** ~1200 LOC + 800 LOC tests per backend.

## kit-new template growth

**Why deferred:** The `kit-new` generator currently stamps one
baseline service shape. Adding template variants for the wave
174 reference patterns (webhook-receiver, background-worker,
realtime-broadcast, api-gateway, saga-coordinator) requires
both new template trees AND generator logic to route between
them — a substantial template-language + integration-test
project.

**Scope:**

- New template trees under
  `cmd/kit-new/templates/<pattern>/` — at minimum:
  - `webhook-receiver/` — HMAC signedrequest + idempotency
    middleware chain, persistent Redis idempotency store
    wiring, secret-manager KeyResolver stub.
  - `background-worker/` — TypedSubscription + circuitbreaker
    + retry, real backend selection prompt
    (AMQP / Kafka / NATS / Redis), lifecycle.Runner wiring.
  - `api-gateway/` — full app.Builder composition with JWT,
    rate limit, downstream gRPC fan-out, OpenAPI exposed.
- Generator changes (`cmd/kit-new/scaffold.go`):
  - New `--template <pattern>` flag with enumerated values.
  - Per-template prompts (backend choice for worker, key store
    choice for receiver).
  - Per-template README + Make targets + smoke test stubs.
- Integration tests
  (`cmd/kit-new/scaffold_test.go`): one per template,
  asserts the generated tree builds and the smoke test passes.

**Estimated size:** ~800 LOC of template strings + ~400 LOC
of generator wiring + ~600 LOC of integration tests.

## What's NOT here

These were considered for v2.2+ during the wave 170–175 sweep
and intentionally rejected:

- **Additional KMS adapters** beyond the four shipped. The
  user explicitly capped the v2.0.0 scope at the existing set
  (`awskms`, `gcpkms`, `azurekeyvault`, `vaulttransit`).
  Additional adapters require concrete consumer demand before
  scoping.
- **A built-in saga durability layer** that wraps outbox +
  idempotency + advisory lock. The composition recipe in
  `examples/README.md` captures the pattern; absorbing it into
  `runtime/saga` directly is a v3 design discussion (the
  surface that lands first will shape every downstream
  service's saga shape).
- **HTTP/3 support in httpx.** Quic stack is still maturing
  across the Go HTTP ecosystem; revisit when the stdlib lands
  a stable surface.

## Picking up an item

A future session can resume any of these by:

1. Reading this doc and the v2.0.0 release notes.
2. Running `make check-doc-rot` to confirm baseline cleanliness.
3. Following the per-package `AGENTS.md` for the new module's
   neighbours (e.g. `infra/messaging/natsbackend/AGENTS.md`
   is the closest reference for SQS / Pub/Sub).
4. Wiring the new module into `go.work`, the dependency
   allowlist, the operational-readiness matrix, the API-freeze
   coverage matrix, and the dashboards + runbooks pack — all of
   the wave 170 / 172 surface applies.
