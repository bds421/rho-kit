# io/ — atomicfile, progress

### [HIGH] Atomic file write loses original permissions and ownership
**File**: `io/atomicfile/atomicfile.go:44,67`
**Issue**: `os.CreateTemp` creates the temp file with mode `0600`. `os.Rename` overwrites the target preserving the new inode → existing file with mode `0644` (or chmod'd, or chown'd) is silently replaced with `0600` and the calling user as owner. Group-readable config files become unreadable to other processes after a save.
**Fix**: Stat the existing target before write; chmod the temp file to match before rename. Document the behavior.
**Effort**: S
**Phase**: 3

### [HIGH] Atomic file `os.Rename` doesn't handle `EXDEV`
**File**: `io/atomicfile/atomicfile.go:67`
**Issue**: When temp and target are on different filesystems (target on mounted volume, `TMPDIR` on tmpfs — common in containers), `rename(2)` returns `EXDEV` and the save fails. Temp file is in same dir as target so this is rare in practice, but the parent-dir-as-bind-mount case fails.
**Fix**: On `EXDEV`, fall back to `io.Copy + Sync + Close + Remove(temp)`. Or document that target directory must be on a single filesystem.

### [LOW] `progressReader`/`throttledReader` not safe for concurrent reads
**File**: `io/progress/progress.go:34-41` + `io/progress/throttle.go:70-127`
**Issue**: `bytesSent`/`lastTime` mutated non-atomically. `io.Reader` is conventionally non-concurrent so this matches convention but the package never says so.
**Fix**: Add doc comment "not safe for concurrent Read".

### [LOW] `progressReader` calls callback on every Read
**File**: `io/progress/progress.go:34-41`
**Issue**: `Read(8KB)` from a stream → callback per 8KB. Multi-GB transfer = hundreds of thousands of callbacks; CPU cost can dominate the transfer.
**Fix**: Add `WithThrottle(d time.Duration)` or `WithMinDelta(bytes int64)` to coalesce callbacks.
