# AGENTS.md — `httpx/openapigen`

## When to use this package

- Service uses kit typed handlers (`httpx.Handle[Req, Resp]`-style) and wants an OpenAPI 3.1 document generated from those types automatically.
- The document must be served from the same process — `Spec.Handler()` returns an `http.Handler` mounted at `/openapi.json` (or wherever you choose).
- Want to use the kit's existing JSON Schema generation (`validate.SchemaFor`) so request/response schemas in the document match the runtime validation.

## When to use something else

- **OpenAPI document is hand-written or generated by a separate tool** — `httpx/openapi` adapts oapi-codegen's StrictServer pattern instead.
- **Need a multi-spec workflow (multiple OpenAPI files)** — out of scope; this is a single-spec generator.

## Key APIs

- `NewSpec(title, version)` — both required.
- `Register(method, path, opts...)` — low-level registration; mostly use the typed `Handle*` helpers from `handle.go` which register on both mux AND spec.
- `Handle[Req, Resp]`, `HandleStatus[Req, Resp]`, `HandleNoBody[Resp]`, `HandleNoBodyStatus[Resp]`, `HandleNoContent` — wave-128 typed registrations.
- Options:
  - `WithTags`, `WithSummary`, `WithDescription`, `WithOperationID`, `WithDeprecated`, `WithSecurity`
  - `WithParameter(p)` — explicit declaration (overrides wave-162 auto-discovery)
  - `WithParameterExample(name, example)` — attaches an example to a parameter by name
  - `WithExternalDocs(url, description)` — per-operation externalDocs link
  - `WithRequestExample(example)` / `WithResponseExample(status, mediaType, example)` — wave-163 example payloads
  - `WithResponseContentT[T](status, mediaType)` / `WithResponseContent(status, mediaType, schema)` — wave-161 multiple content types per status
  - `WithResponseHeader(status, name, header)` — wave-161 response headers
  - `WithSkipPathParamDiscovery()` — disable wave-162 auto-discovery if you want every parameter to be code-review visible

## Common mistakes

- **Re-declaring auto-discovered path parameters via WithParameter without richer schema** — auto-discovery already emits required string-typed parameters from the `{name}` template. Only override when you have a tighter schema (UUID format, enum constraint).
- **Skipping `WithRequestExample` / `WithResponseExample`** — generated portals show the schema but no concrete payload. Adding examples is a 1-line improvement that makes the docs dramatically more useful.
- **Embedding response headers in the response body** — use `WithResponseHeader(status, name, header)` so portals render them in the right OpenAPI section.
- **`Spec.Handler()` mounted publicly without intent** — exposing the spec is a deployment decision. Some teams hide it behind auth; others publish openly. The kit does NOT mount it for you.

## Observability

- Spec rendering is in-memory cached; mutations invalidate the cache. No metrics emitted.
